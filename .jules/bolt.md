## 2026-04-10 - [IndexedDB Query Optimization]
**Learning:** O(N) full table scans were happening in `core.queryAtoms` because it was fetching `storage.getAllAtoms()` and filtering in memory, despite the secondary indices (`contentHash` and `field`) being defined in the `storage` initialization scheme. Opening individual batched reads is slower, but querying by index is extremely fast.
**Action:** When querying IndexedDB frontend storage (`src/attrstrand/storage.ts`), avoid performing a single `getAll()` scan and memory-filtering it if you can use `db.getAllFromIndex` for queries involving secondary indices like `contentHash` or `field`.
